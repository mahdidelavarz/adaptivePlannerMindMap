# Discussion 021 — Validation Plan and Decision Gates

## Status

Open for GPT × Claude review.

## Purpose

Define how the first pilot evaluates AI-assisted creation and AI-assisted Reconcile using predeclared descriptive thresholds, explicit denominators, qualitative evidence, and decision gates without overstating causality.

This discussion does not ask whether AI is impressive. It asks whether the bounded product loops accepted in Discussions 012–020 produce sufficiently useful, understandable, and trustworthy outcomes to justify continuation.

---

## 1. Governing Principles

1. Pilot results are descriptive unless a design explicitly supports causal inference.
2. Thresholds, denominators, exclusions, segmentation rules, and decision logic must be locked before pilot outcome data is reviewed.
3. Every rate must name its denominator.
4. Exposure, attempt, valid output, reviewable proposal, confirmation, and applied command are different stages and must not be collapsed.
5. Edited AI output remains AI-assisted but must be measured separately from unchanged acceptance.
6. Operational reliability failures must not be misreported as user rejection.
7. Low-volume severity bands remain inconclusive rather than being merged opportunistically after results are seen.
8. Self-reported usefulness is evidence, not proof of durable value.
9. Safety, privacy, or trust regressions override positive aggregate productivity metrics.
10. No decision gate may authorize weakening accepted guardrails merely to improve conversion.

---

## 2. Formal Hypotheses

### H1 — AI-Assisted Creation

Users can reach a useful, credible first plan through bounded AI assistance.

Operationally, H1 requires evidence across four dimensions:

- **Reachability:** users can obtain a reviewable PlanningDraft.
- **Usefulness:** the draft leads to a plan the user considers worth keeping.
- **Credibility:** users understand that the draft is provisional and trust the confirmation boundary.
- **Execution relevance:** the resulting plan contains non-trivial actionable structure rather than decorative or trivial output.

### H2 — AI-Assisted Reconcile

AI can reduce unresolved work into fewer understandable, user-approved decisions without replacing deterministic Reconcile facts or mutation authority.

Operationally, H2 requires evidence across four dimensions:

- **Compression:** unresolved items are reduced into fewer recommendation decisions.
- **Comprehension:** users understand why recommendations were shown.
- **Resolution:** users approve, edit, reject, defer, or manually resolve recommendations.
- **Trust:** accepted recommendations do not create confusion about whether commands actually succeeded.

## 3. Unit Definitions

### 3.1 Eligible User

A pilot participant who:

- has completed required onboarding,
- has accepted pilot consent and instrumentation disclosures,
- is not an internal team member or close collaborator unless separately labeled,
- has sufficient product data to encounter at least one eligible H1 or H2 flow.

Internal, demo, QA, seeded, and synthetic accounts are excluded from primary pilot metrics and reported separately.

### 3.2 Exposure

An exposure occurs only when the user is shown an eligible entry point and the required deterministic prerequisites are available.

An exposure is not counted when:

- the feature is disabled by kill switch,
- the user lacks permission,
- required canonical data is unavailable,
- the runtime rejects the request before an eligible flow can begin,
- the event was generated by test automation or internal accounts.

### 3.3 Attempt

An attempt begins when a valid user action creates a logical PlanningAttempt or ReconcileSession.

Duplicate client retries with the same idempotent identity remain one logical attempt.

### 3.4 Valid AI Outcome

A valid AI outcome is a result that passes all accepted 020C validation gates.

Provider responses that are partial, invalid, repaired unsuccessfully, cancelled, timed out, over budget, blocked by safety policy, or discarded after cancellation are not valid outcomes.

### 3.5 Reviewable Outcome

A valid outcome persisted and presented to the user for review.

### 3.6 Confirmed Action

A user explicitly submits an ActionConfirmation after viewing the current server-authoritative preview.

Confirmation does not mean mutation success.

### 3.7 Applied Command

A deterministic CommandResult reports successful canonical mutation.

---

## 4. Metric Classes

Metrics are labeled as:

- **Behavioral:** derived from user actions and state transitions.
- **Self-reported:** captured through in-product questions or interviews.
- **Operational:** runtime, validation, reliability, cost, and latency measurements.
- **Safety/Trust:** guardrail, confusion, override, regret, and harmful-output indicators.

No single class is sufficient to pass a hypothesis.

---

## 5. H1 Metric Dictionary

### H1-M1 — Eligible Attempt Rate

```txt
eligible users who start at least one PlanningAttempt
/
eligible users exposed to AI-assisted creation
```

Behavioral. Describes willingness to try, not usefulness.

### H1-M2 — Reviewable Draft Rate

```txt
PlanningAttempts producing a reviewable PlanningDraft
/
valid logical PlanningAttempts started
```

Operational-behavioral. Must also report failure-class distribution.

### H1-M3 — Draft Review Completion Rate

```txt
reviewable PlanningDrafts receiving an explicit user disposition
/
reviewable PlanningDrafts shown
```

Explicit dispositions:

- accept unchanged,
- accept after edit,
- reject,
- abandon after meaningful review,
- expire without review.

### H1-M4 — Unchanged Acceptance Rate

```txt
reviewable PlanningDrafts confirmed without semantic edit
/
reviewable PlanningDrafts with explicit disposition
```

This is not the main success metric. A high value may indicate fit, passivity, trivial output, or insufficient review.

### H1-M5 — Edited Acceptance Rate

```txt
reviewable PlanningDrafts confirmed after semantic edit
/
reviewable PlanningDrafts with explicit disposition
```

Edited acceptance counts as AI-assisted success but is reported separately.

Semantic edits include changes to:

- entity creation/removal,
- ownership or parent relationships,
- dates or recurrence,
- priority or scope,
- warnings or protected consequences,
- action type.

Cosmetic edits do not move an outcome into edited acceptance.

### H1-M6 — Applied Plan Rate

```txt
PlanningDrafts whose linked confirmation produces successful CommandResult
/
reviewable PlanningDrafts with explicit disposition
```

Also report confirmed-but-conflicted and confirmed-but-failed separately.

### H1-M7 — Useful First Plan Rate

A plan is counted as useful only when all conditions hold:

1. the user explicitly rates the result as at least useful on the locked pilot scale,
2. the linked command succeeds,
3. the resulting plan survives the minimum observation window without full deletion or immediate reversal,
4. the result passes the non-triviality rule in section 7.

```txt
useful first plans
/
reviewable PlanningDrafts with explicit disposition
```

### H1-M8 — Time to Reviewable Draft

Measured from logical attempt creation to reviewable outcome.

Report median, p75, p90, timeout count, and cancellation count. Mean alone is forbidden because latency distributions enjoy lying when given the opportunity.

### H1-M9 — Trust Boundary Comprehension

Percentage of sampled users who correctly answer that:

- AI output is a proposal,
- confirmation is required,
- confirmation does not prove command success,
- canonical changes come from deterministic command handlers.

Measured through a short pilot comprehension check and interviews.

### H1-M10 — Immediate Regret or Reversal Rate

```txt
successfully applied plans materially reversed or deleted inside the locked regret window
/
successfully applied AI-assisted plans
```

Must distinguish correction of user intent from ordinary later evolution.

---

## 6. H2 Metric Dictionary

### H2-M1 — Eligible Reconcile Start Rate

```txt
eligible ReconcileSessions started
/
eligible Reconcile exposures
```

### H2-M2 — Deterministic Reconcile Availability Rate

```txt
sessions producing valid deterministic FACT and RULE_MATCH output
/
valid ReconcileSessions started
```

AI explanation eligibility is measured separately.

### H2-M3 — Explanation Availability Rate

```txt
sessions receiving valid AI_EXPLANATION
/
sessions eligible for AI explanation
```

This must not be used to hide deterministic Reconcile failures.

### H2-M4 — Decision Compression Ratio

```txt
number of unresolved source items entering Reconcile
/
number of recommendation decisions shown
```

Report distribution and severity band. Higher is not automatically better because over-compression can hide materially distinct decisions.

### H2-M5 — Recommendation Disposition Rate

```txt
recommendations receiving explicit disposition
/
recommendations shown
```

Disposition categories:

- accepted unchanged,
- accepted after edit,
- rejected,
- deferred,
- manually resolved outside recommendation,
- abandoned.

### H2-M6 — Recommendation Acceptance Rates

Unchanged and edited acceptance are reported separately using recommendations with explicit disposition as denominator.

Recommendation `outcome=ACCEPTED` is not command success.

### H2-M7 — Applied Recommendation Rate

```txt
accepted recommendations whose resulting command succeeds
/
recommendations with explicit disposition
```

Also report conflicted and failed command results.

### H2-M8 — Unresolved Work Reduction

```txt
eligible unresolved items before session
-
eligible unresolved items remaining after locked observation point
```

Report absolute and percentage reduction. Items newly created after session start are not credited or blamed unless explicitly linked to the session.

### H2-M9 — Reconcile Understanding Score

Self-reported score plus interview evidence for:

- why an item appeared,
- what action was proposed,
- what consequence would follow,
- whether the action actually succeeded.

### H2-M10 — Reopen or Regret Rate

```txt
applied Reconcile actions materially reversed or reopened within the locked regret window
/
successfully applied Reconcile actions
```

### H2-M11 — Manual Escape Success

Percentage of failed, degraded, or rejected AI-assisted sessions in which the user successfully continues through deterministic or manual paths.

This guards against a superficially good AI flow that turns failure into a dead end.

## 7. Non-Triviality and Usefulness Rules

A Planning result is not considered useful merely because it was accepted.

A result is non-trivial only when it creates or materially updates at least one meaningful execution structure, such as:

- a Goal with at least one actionable child,
- a Project with coherent actionable work,
- multiple linked Tasks with valid ownership,
- a Routine with valid recurrence and execution relevance,
- a substantive restructuring of existing work.

The following do not independently satisfy non-triviality:

- one generic task with no meaningful context,
- restatement of the user's input,
- cosmetic renaming,
- empty containers,
- plans immediately deleted before use,
- structures accepted only to exit the flow.

A deterministic non-triviality classifier must use canonical structure and events, not an AI judgment over raw text.

## 8. Event-to-Metric Mapping

The pilot analysis must define exact event inputs for every metric using accepted 019C semantics.

Minimum event families:

- eligible exposure recorded,
- PlanningAttempt started/completed/failed/cancelled,
- PlanningDraft reviewable/superseded/expired/cancelled,
- draft semantic edit recorded,
- ActionConfirmation created/submitted/expired,
- CommandResult succeeded/conflicted/failed,
- ReconcileSession started/completed/degraded/unavailable,
- recommendation shown/dispositioned,
- deterministic facts available/unavailable,
- AI explanation available/unavailable,
- canonical mutation and reversal events,
- self-report submitted,
- interview sample linked through privacy-safe participant identity.

Every metric specification must include:

```txt
metric name
unit of analysis
numerator
 denominator
eligible population
exclusions
observation window
required event fields
segmentation fields
missing-data behavior
```

Generic `*_UPDATED` events without required changedFields remain ineligible where semantic edit classification is needed.

## 9. Failure and Abandonment Classification

Outcomes must distinguish:

- user rejection,
- user cancellation,
- user abandonment before review,
- user abandonment after meaningful review,
- provider failure,
- validation failure,
- safety-policy block,
- budget/rate-limit block,
- deterministic Reconcile failure,
- stale-version conflict,
- command failure,
- product or transport defect.

Operational failure is never counted as user rejection.

A session is classified as abandoned only after the locked inactivity window and only when no terminal user or system state exists.

---

## 10. Severity Segmentation for Reconcile

Every ReconcileSession is assigned one immutable severity band from deterministic pre-session facts.

### Light

Small number of unresolved items, no protected conflict, no major deadline risk, and actions remain independently understandable.

### Medium

Multiple unresolved items, dependencies or deadline tension, but no Recovery-level condition.

### Recovery

Substantial backlog, repeated misses, conflicting commitments, expired structures, major deadline risk, or conditions requiring staged recovery rather than routine cleanup.

The exact classifier and thresholds must be versioned before pilot analysis.

Severity may not be reclassified after observing session outcome.

Metrics are reported overall and by severity, but a severity band receives a decision only after meeting its own minimum exposure rule.

---

## 11. Exposure and Interpretability Rules

This pilot is not powered for formal causal inference. Minimums exist to prevent decisions from being made on anecdotes wearing spreadsheet clothing.

### 11.1 Overall Hypothesis Minimum

A hypothesis is decision-eligible only when all are met:

- at least 30 eligible users exposed,
- at least 20 valid logical attempts or sessions,
- at least 15 reviewable outcomes,
- at least 10 explicit dispositions,
- no single user contributes more than 10% of attempts in the primary analysis.

These are minimum interpretability rules, not statistical power claims.

### 11.2 Severity-Band Minimum

A Reconcile severity band is decision-eligible only with:

- at least 10 eligible sessions,
- at least 7 reviewable recommendation sets,
- at least 5 explicit user dispositions,
- participation from at least 5 distinct users.

Otherwise the band is marked inconclusive and reported descriptively.

### 11.3 Repeated Exposure

Primary user-level metrics count one user's first eligible exposure within the locked evaluation window.

Attempt-level metrics may include repeated exposure but must report:

- first attempt,
- repeat attempts,
- per-user clustered distribution.

Repeated attempts must not make one enthusiastic user look like a population.

### 11.4 Missing Data

Missing self-report data is not imputed as positive or negative.

Behavioral and operational metrics remain reportable with explicit missingness.

## 12. Proposed Predeclared Thresholds

Thresholds below are pilot decision thresholds, not universal product truths.

### 12.1 H1 Threshold Set

#### Strong Success

All must hold:

- Reviewable Draft Rate ≥ 75%
- explicit Draft Review Completion Rate ≥ 70%
- Useful First Plan Rate ≥ 50%
- Applied Plan Rate ≥ 55%
- Trust Boundary Comprehension ≥ 80%
- Immediate Regret/Reversal Rate ≤ 15%
- no unresolved safety or privacy regression

#### Weak Success

All must hold:

- Reviewable Draft Rate ≥ 60%
- Draft Review Completion Rate ≥ 55%
- Useful First Plan Rate ≥ 35%
- Applied Plan Rate ≥ 40%
- Trust Boundary Comprehension ≥ 70%
- Immediate Regret/Reversal Rate ≤ 25%
- no unresolved safety or privacy regression

#### Failure

Any of:

- Useful First Plan Rate < 25%
- Applied Plan Rate < 30%
- Trust Boundary Comprehension < 60%
- Immediate Regret/Reversal Rate > 30%
- repeated qualitative evidence that plans are trivial, misleading, or harder than manual creation
- material safety, privacy, or trust regression

#### Inconclusive

- minimum exposure rules not met,
- key event quality is invalid,
- missingness prevents interpretation,
- results fall between weak success and failure boundaries without coherent qualitative explanation.

### 12.2 H2 Threshold Set

#### Strong Success

All must hold:

- Deterministic Reconcile Availability Rate ≥ 95%
- recommendation explicit disposition rate ≥ 70%
- Applied Recommendation Rate ≥ 50%
- median unresolved-work reduction ≥ 40%
- Reconcile Understanding Score reaches locked positive threshold for ≥ 75% of respondents
- Reopen/Regret Rate ≤ 15%
- Manual Escape Success ≥ 90%
- no unresolved safety or privacy regression

#### Weak Success

All must hold:

- Deterministic Reconcile Availability Rate ≥ 90%
- recommendation disposition rate ≥ 55%
- Applied Recommendation Rate ≥ 35%
- median unresolved-work reduction ≥ 25%
- positive understanding for ≥ 65% of respondents
- Reopen/Regret Rate ≤ 25%
- Manual Escape Success ≥ 80%
- no unresolved safety or privacy regression

#### Failure

Any of:

- Deterministic Reconcile Availability Rate < 85%
- Applied Recommendation Rate < 25%
- median unresolved-work reduction < 15%
- positive understanding < 55%
- Reopen/Regret Rate > 30%
- repeated evidence of over-compression or misunderstood consequences
- AI failure blocks deterministic/manual continuation
- material safety, privacy, or trust regression

#### Inconclusive

Same interpretability rules as H1, including insufficient severity-band exposure.

## 13. Operational Guardrail Gates

Positive H1 or H2 metrics cannot produce a pass when any hard guardrail gate fails.

Hard gates:

- confirmed canonical mutation without valid confirmation,
- partial or invalid output becoming reviewable,
- tool/function calling enabled against accepted 020A policy,
- provider/model fallback occurring in pilot,
- raw sensitive content retained outside accepted policy,
- crisis or restricted events used for ordinary analytics,
- idempotency defect causing duplicate canonical mutation,
- cancellation result becoming visible after cancellation,
- provider spend cap bypass,
- unresolved severe authorization or ownership breach.

Any hard-gate failure pauses the affected capability until corrected and revalidated.

## 14. Qualitative Evidence Plan

Quantitative events are accompanied by:

### 14.1 Moderated Observation

Observe at least:

- 5 H1 users,
- 5 H2 users,
- representation across available severity bands,
- at least 3 users who reject or abandon an AI result where available.

Observe:

- expectation before generation,
- comprehension of provisional status,
- warning and consequence review,
- editing behavior,
- confirmation comprehension,
- reaction to conflicts and failures,
- ability to continue manually.

### 14.2 Semi-Structured Interviews

Interview prompts cover:

- what the user believed AI was doing,
- what made the output useful or untrustworthy,
- whether edits felt easier than manual creation,
- whether recommendations reduced cognitive load,
- whether any consequence was surprising,
- whether they knew if the command succeeded,
- what they would do without AI.

### 14.3 Evidence Coding

Qualitative findings use a predeclared codebook:

- useful structure,
- trivial restatement,
- missing context,
- incorrect assumption,
- warning comprehension,
- confirmation confusion,
- command-status confusion,
- over-compression,
- trust increase,
- trust decrease,
- manual fallback success,
- abandonment cause.

New codes may be added during analysis but must be labeled emergent rather than silently folded into predeclared themes.

---

## 15. Comparison Rules

The following comparisons are descriptive only unless a later study introduces valid assignment or matching:

- manual vs AI-assisted creation,
- AI explanation available vs degraded facts-only Reconcile,
- first-time vs repeat users,
- Light vs Medium vs Recovery,
- edited vs unchanged acceptance,
- provider configuration or prompt versions,
- users who choose AI vs users who do not.

Reports must not imply that observed differences were caused by AI.

Selection bias, learning effects, user experience, data complexity, and self-selection must be named as alternative explanations.

---

## 16. Forbidden Claims

Pilot results must not be described as proving that AI:

- increases productivity,
- improves mental health,
- reduces stress,
- builds habits,
- causes better completion rates,
- is superior to manual planning,
- is safe for all users or domains,
- generalizes beyond the pilot population,
- works equally across severity bands,
- replaces professional advice,
- creates durable behavior change,
- explains differences between cohorts without causal design.

Permitted language:

- “In this pilot, X% of eligible attempts produced…”
- “Participants reported…”
- “Observed outcomes were consistent/inconsistent with the hypothesis…”
- “The pilot provides descriptive evidence to continue, narrow, redesign, or stop…”

---

## 17. Threshold Locking Procedure

Before the first pilot outcome is reviewed:

1. freeze metric definitions,
2. freeze event queries or versioned analysis code,
3. freeze exclusions and denominator rules,
4. freeze severity classifier version,
5. freeze observation windows,
6. freeze threshold table and decision matrix,
7. record document commit SHA and analysis package version,
8. record who approved the lock and timestamp,
9. separate pilot-monitoring dashboards from decision-analysis outputs.

After lock, changes require:

- a written amendment,
- stated reason unrelated to observed outcome preference,
- versioned before/after definitions,
- analysis under both definitions where feasible,
- explicit labeling as post-lock change.

Thresholds may not be tuned because results were inconvenient. Numbers do not become more scientific merely because the team negotiates with them after lunch.

## 18. Decision Matrix

### H1 Strong + H2 Strong

Decision:

- continue both capabilities into controlled rollout planning,
- keep all accepted guardrails,
- use Discussion 022 for rollout thresholds and operational readiness.

### H1 Strong/Weak + H2 Failure

Decision:

- continue or iterate AI-assisted creation,
- pause AI explanation/recommendation expansion for Reconcile,
- preserve deterministic Reconcile,
- diagnose comprehension, compression, and command-result failures.

### H1 Failure + H2 Strong/Weak

Decision:

- preserve Reconcile path,
- redesign or narrow Planning scope,
- test smaller planning jobs, better context collection, or manual-first creation.

### Both Weak Success

Decision:

- no broad rollout,
- run one bounded iteration focused on the dominant shared failure modes,
- retain pilot-only exposure.

### Either Inconclusive, Other Positive

Decision:

- continue only the evidence-supported capability,
- collect additional exposure for the inconclusive hypothesis without relaxing thresholds.

### Either Failure with Hard Guardrail Failure

Decision:

- pause affected AI capability,
- remediate and revalidate before further pilot exposure.

### Both Failure

Decision:

- stop expansion,
- preserve manual and deterministic product value,
- revisit the AI thesis, user problem, and scope rather than polishing prompts indefinitely.

## 19. Scope Reduction and Redesign Triggers

Reduce scope when:

- valid output is useful only for a narrow entity or user segment,
- Recovery severity performs materially worse with adequate exposure,
- context floor frequently exceeds budget,
- edits cluster around one unsupported action type,
- users reliably accept only small bounded proposals.

Redesign UX when:

- users confuse draft review, confirmation, and command success,
- warnings are acknowledged without understanding,
- abandonment occurs during consequence review,
- users cannot recover after conflict or failure,
- recommendation compression obscures distinct decisions.

Revisit thesis when:

- useful outcomes remain below failure threshold after one bounded redesign,
- manual paths consistently outperform in observed usability without compensating benefit,
- users report that AI adds review burden rather than reducing it,
- trust or safety costs outweigh measured value.
